**CURRENT STATE:**
A especificação `architecture/02-technical/api-contracts/proposal-api.v1.yaml` está funcional, mas desalinhada com três dos nossos principais ADRs.

**EXPECTED (Estado Final de Sucesso):**
O arquivo `architecture/02-technical/api-contracts/proposal-api.v1.yaml` é modificado para refletir as seguintes mudanças mandatórias:

1.  **Conformidade com ADR-007 (Versionamento):** Todos os `paths` da API devem ser atualizados para incluir o prefixo de versão `/v1`. (Ex: `/propostas/{id}` deve tornar-se `/v1/propostas/{id}`).
2.  **Conformidade com ADR-004 (Tratamento de Erros):** Todas as respostas de erro (`4xx`, `5xx`) em todos os endpoints devem ser refatoradas para usar um `$ref` que aponte para um schema `ProblemDetails` central, em conformidade com a **RFC 7807**.
3.  **Conformidade com ADR-008 (Segurança de PII):** Os schemas de resposta (`ProposalResponse`, etc.) para os endpoints `GET` devem ser refatorados. Eles **NÃO DEVEM** conter campos de PII em texto plano. Em vez disso, os campos como `clienteCpf`, `clienteRg`, etc., devem ser atualizados para refletir o seu formato mascarado (ex: `type: string`, `example: "***.***.123-**"`).

**CONSTRAINTS (Roadmap de Implementação):**
1.  **Refatore o Versionamento:** Adicione o prefixo `/v1` a todos os `paths`.
2.  **Refatore os Erros:** Crie um schema reutilizável `ProblemDetails` na seção `components/schemas` e refatore todas as respostas de erro para o utilizar.
3.  **Refatore a Segurança de PII:** Modifique os schemas de resposta para refletir a nossa política de mascaramento de dados.
