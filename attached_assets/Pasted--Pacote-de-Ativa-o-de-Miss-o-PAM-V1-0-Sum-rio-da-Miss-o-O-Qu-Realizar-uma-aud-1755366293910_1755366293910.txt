**### Pacote de Ativação de Missão (PAM) V1.0**

Esta auditoria nos dará o mapa exato das falhas para que possamos construir uma solução definitiva e confiável.
* **Histórico Relevante (Consulta Obrigatória):** A diretriz do Arquiteto (Blueprint V4.0) é nossa nova fonte da verdade. Ignore todas as implementações anteriores. Sua missão é avaliar o código atual estritamente contra estas novas regras.
* **Modelo Mental (Como se Encaixa):** Você é um auditor de sistemas. O Blueprint V4.0 é o seu checklist de conformidade. Sua tarefa é ler, analisar e relatar as discrepâncias entre a especificação e a realidade do código.
* **Riscos Antecipados:** **Risco:** Uma análise superficial levará a uma correção incompleta. **Contramedida:** A sua missão não é resolver, mas sim **PROVAR**, com evidências de código, onde o sistema falha.

**CONSTRAINTS (Roadmap de Auditoria Forense):**

**1. Auditoria dos Botões "Copiar PIX":**
    - **Ação:** Inspecione o endpoint que busca os dados para a Ficha do Cliente (`/api/cobrancas/:id/ficha`) e o componente de frontend que renderiza a lista de parcelas.
    - **Relatório:** Por que os botões "Copiar PIX" não funcionam? A query do backend está a falhar em retornar o campo `pixCopiaECola` da tabela `inter_collections`? Ou o frontend está a falhar em receber ou processar este dado? Apresente o `console.log` dos dados de uma parcela que chegam ao componente.

**2. Auditoria do Fluxo de Download do Boleto:**
    - **Ação:** Analise a lógica do botão de download de boletos individuais na Ficha do Cliente.
    - **Relatório:** Qual é o fluxo de eventos atual quando o botão é clicado? Ele chama um endpoint? Que resposta ele recebe? Descreva o comportamento atual.

**3. Auditoria do Status das Parcelas:**
    - **Ação:** Re-examine o endpoint `/api/cobrancas/:id/ficha`.
    - **Relatório:** Para obter o status de cada parcela, o backend está a fazer uma **chamada em tempo real** para a API do Banco Inter, ou está a ler os dados **potencialmente desatualizados** da nossa tabela local `inter_collections`? Apresente o código que busca o status como evidência.

**4. Auditoria da Funcionalidade "Marcar como Pago":**
    - **Ação:** Faça uma busca no código do frontend e do backend.
    - **Relatório:** Existe alguma implementação, mesmo que parcial ou em placeholder, para um operador de cobranças marcar manualmente uma parcela como paga? Se não, declare `[FUNCIONALIDADE AUSENTE]`.

**Relatório Final:**
O critério de sucesso é um relatório que nos dê um mapa claro de todas as correções necessárias para alinhar 100% a "Ficha do Cliente" com o nosso novo Blueprint de Negócio.
