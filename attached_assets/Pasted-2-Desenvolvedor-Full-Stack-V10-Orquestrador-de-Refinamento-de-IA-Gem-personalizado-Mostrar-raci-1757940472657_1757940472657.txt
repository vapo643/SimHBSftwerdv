Análise Forense da Causa Raiz (RCA)
Os logs apresentam um padrão claro e inequívoco de um loop de autenticação infinito:

Token Válido, Verificação Inválida: O frontend inicia ([AUTH EVENT] SIGNED_IN), obtém um token da Supabase e tenta fazer uma chamada de API (GET /api/debug/me).

Rejeição do Backend (401): Nosso backend recebe o token, mas o rejeita como inválido (Token inválido ou expirado).

Ciclo de Refrescar e Falhar: O frontend, ao receber o 401, corretamente invalida o token e solicita um novo token à Supabase (✅ [TOKEN MANAGER] Token refreshed successfully). No entanto, a chamada de API subsequente com este novo e recém-gerado token falha novamente com um 401.

Veredito Irrefutável: O problema não está na geração do token, mas na verificação. Nosso backend está a usar um segredo JWT (JWT_SECRET ou SUPABASE_JWT_SECRET) que é diferente do segredo que a Supabase está a usar para assinar os tokens que entrega ao cliente. O resultado é que cada token, por mais válido que seja do ponto de vista da Supabase, é considerado uma falsificação pelo nosso backend.

Esta é uma falha de configuração de ambiente de Nível 1.
