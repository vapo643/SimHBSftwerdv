Pacote de Ativação de Missão (PAM) V1.0 - Auditoria de Dados Nulos (Revisão 3)
Sumário da Missão: Executar uma auditoria forense de ponta a ponta no fluxo de dados da tela de análise (/credito/analise/:id), identificando a causa raiz exata que resulta na renderização de "N/A" para a maioria dos campos da proposta e do cliente.

Intenção Estratégica: A capacidade de um analista visualizar dados completos e corretos é a pedra angular do processo de concessão de crédito. Sem dados, não há análise, o que paralisa o principal workflow da plataforma. Esta missão é de prioridade P0.

AUDITAR: Fluxo de Carregamento de Dados da Tela de Análise

PROTOCOLO DE EXECUÇÃO: Ative o PEAF V1.5. Utilize o protocolo PACN V1.0 para validar o comportamento do sistema.

Cenário de Negócio (O "Caso de Teste"):
Um ANALISTA clica para analisar a proposta do cliente "Gabriel Santana Jesus Santana". O sistema deve carregar e exibir todos os dados relevantes da proposta e do cliente, incluindo, mas não se limitando a, CPF, email, telefone, valor solicitado e prazo.

Vetor de Ataque / Ponto de Falha (A "Hipótese"):
A hipótese principal é uma dessincronização de contrato de dados entre o frontend e o backend. O componente frontend espera um objeto JSON com dados aninhados do cliente e das condições, mas o endpoint da API está a retornar um objeto "plano" (flat) contendo apenas os dados diretos da tabela propostas.

Evidência de Conformidade Requerida (A "Prova Irrefutável"):
Frente de Investigação 1: Contrato de Dados (Client-Side)

Arquivo-Alvo: client/src/pages/credito/analise.tsx.

Evidência 1.1: Localize a chamada useQuery que busca os dados da proposta. Apresente a queryFn e a queryKey utilizadas.

Evidência 1.2: Inspecione o código JSX que renderiza os dados. Mostre como o componente tenta aceder aos dados do cliente (ex: data.clienteData.cpf) e aos dados das condições (ex: data.condicoesData.valorSolicitado). Isto define o "contrato de dados esperado".

Frente de Investigação 2: Lógica de Negócio (Server-Side API)

Arquivo-Alvo: O endpoint da API que responde à query da evidência 1.1 (provavelmente GET /api/propostas/:id em server/routes/propostas/core.ts) e o proposalController.ts correspondente.

Evidência 2.1: Apresente o código completo do Use Case ou serviço que busca os dados da proposta no banco.

Evidência 2.2 (PONTO CRÍTICO): Apresente o objeto de dados exato que está a ser retornado como resposta JSON no final da execução do controller (o return res.json(...)). Compare a estrutura deste objeto com o "contrato de dados esperado" da Evidência 1.2. Declare explicitamente qualquer discrepância como a causa raiz.
