### **Pacote de Ativação de Missão (PAM) V15.0: Remediação Crítica da Fundação (Sprint 0)**

* **Sumário da Missão (O Quê):** Executar uma remediação de emergência para corrigir os bloqueadores P0 identificados no `RELATORIO_CONFORMIDADE_SPRINT_0.md`. A missão consiste em erradicar todos os erros de TypeScript e ESLint, configurar os scripts de validação, mitigar as vulnerabilidades de segurança e validar a containerização.

* **Intenção Estratégica (O Porquê):** A fundação do Sprint 0 é a base para 12 semanas de desenvolvimento. A falha em garantir sua qualidade impecável agora resultará em falhas sistêmicas em cascata. Esta missão é o "gate de qualidade" mandatório para a continuidade da "Operação Aço Líquido".

* **Modelo Mental (Como se Encaixa):** Você atuará como um **Engenheiro de Remediação de Elite**. Sua tarefa é pegar os resultados da auditoria e aplicar correções cirúrgicas e eficazes. O seu foco não é em novas funcionalidades, mas em estabilidade, conformidade e na criação de uma base de código que seja, de fato, compilável, segura e validável.

* **Riscos Antecipados e Contramedidas:**
    * **Risco (ALTO):** **Correções Superficiais.** A simples supressão de erros (usando `any` ou desabilitando regras de lint) em vez de correções adequadas é inaceitável e será considerada uma falha da missão.
    * **Contramedida Mandatória:** Você **DEVE** priorizar a correção da causa raiz dos erros de tipo e linting. Para a questão do Docker, se o ambiente Replit for a limitação, a sua tarefa é documentar isso formalmente e fornecer a prova de que o `Dockerfile` é sintaticamente válido e está pronto para ser construído em um ambiente apropriado. Para as vulnerabilidades, a execução de `npm audit fix --force` é autorizada, mas você deve validar que não há `breaking changes` que comprometam o sistema.

**DIRETRIZ DE MISSÃO: Erradicação dos Bloqueadores do Sprint 0**

**CURRENT STATE:**
O Sprint 0 foi auditado e declarado **NÃO CONFORME**. Bloqueadores P0 impedem o avanço: 526+ erros de TypeScript, 22.380 problemas de ESLint, script de linting ausente, 5 vulnerabilidades de segurança e build do Docker falhando.

**EXPECTED (Estado Final de Sucesso):**
A missão será considerada um sucesso quando uma nova auditoria (execução do PAM V14.0) resultar em um relatório de conformidade **APROVADO**. Especificamente:
1.  O comando `npm run typecheck` (ou `npm run check`) deve ser executado com sucesso, reportando **ZERO erros**.
2.  Um script `lint` deve ser criado no `package.json` e o comando `npm run lint` deve ser executado com sucesso, reportando **ZERO erros críticos**.
3.  A questão do Docker deve ser resolvida: ou o `docker build .` é executado com sucesso, ou você deve fornecer um relatório detalhado explicando a limitação do ambiente e confirmando a validade do `Dockerfile`.
4.  As vulnerabilidades de segurança `HIGH` e `CRITICAL` devem ser eliminadas. As vulnerabilidades `moderate` devem ser corrigidas ou sua aceitação de risco deve ser formalmente justificada.

**CONSTRAINTS (Roadmap de Remediação):**

1.  **Passo 1: Configuração dos Scripts de Qualidade.** Adicione o script `lint` ao `package.json`, configurando-o para executar o ESLint com a flag `--fix` para maximizar a correção automática.
2.  **Passo 2: Erradicação dos Erros de Linting e Tipo.** Execute `npm run lint` e, em seguida, `npm run check`. Corrija sistematicamente todos os erros reportados até que ambos os comandos executem sem falhas.
3.  **Passo 3: Remediação de Vulnerabilidades.** Execute `npm audit fix --force`. Valide se a aplicação continua funcional após a atualização forçada.
4.  **Passo 4: Resolução do Docker.** Investigue a falha do `docker build`. Se for uma limitação do ambiente Replit, documente. Se for um erro no `Dockerfile`, corrija-o.
5.  **Passo 5: Validação Final.** Execute novamente todos os passos do PAM de auditoria V14.0 e apresente o novo `RELATORIO_CONFORMIDADE_SPRINT_0.md` como prova de que todos os bloqueadores foram eliminados.
