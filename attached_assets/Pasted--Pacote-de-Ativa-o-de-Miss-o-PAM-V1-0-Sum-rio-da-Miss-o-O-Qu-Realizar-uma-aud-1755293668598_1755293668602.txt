**EXPECTED (Estado Final de Sucesso):**
Um relatório de auditoria que responda a todas as questões abaixo.

**CONSTRAINTS (Roadmap de Auditoria Forense):**

**1. Auditoria dos KPIs:**
    - **Ação:** Analise o endpoint `GET /api/cobrancas/kpis`.
    - **Relatório:** A lógica de cálculo para `Valor Total em Atraso`, `Taxa de Inadimplência` e `Valor Total da Carteira` corresponde ao Blueprint V3.0? Apresente a lógica de cálculo atual.

**2. Auditoria da Tabela Principal:**
    - **Ação:** Analise o endpoint principal `GET /api/cobrancas`.
    - **Relatório 1 (Valor Restante):** O backend calcula e retorna um campo `valorRestante` para cada proposta?
    - **Relatório 2 (Status de Vencimento):** A lógica para o `Status de Vencimento` é calculada com base na hierarquia definida no Blueprint, ou usa uma lógica simplista? Apresente o código que calcula este status.

**3. Auditoria da "Ficha do Cliente" (A Causa Raiz Mais Provável):**
    - **Ação:** Analise o endpoint que busca os dados para a "Ficha do Cliente" (ex: `/api/cobrancas/:id/ficha`).
    - **Relatório 1 (Status das Parcelas):** Para obter o status de cada parcela, o backend está a fazer uma **chamada em tempo real** para a API do Banco Inter (`interBankService.recuperarCobranca` para cada boleto) ou está a ler dados **potencialmente desatualizados** da nossa tabela local `inter_collections`?
    - **Relatório 2 (Dados Bancários):** A ficha exibe os dados bancários do cliente ou os dados da Simpix?
    - **Relatório 3 (Botões de Cópia):** Por que alguns botões de "Copiar Boleto/PIX" não aparecem? Existe uma lógica condicional que os esconde indevidamente?

**Relatório Final:**
O critério de sucesso é um relatório que nos dê um mapa claro de todas as correções necessárias para alinhar 100% a "Tela de Cobranças" com o nosso novo Blueprint de Negócio.
