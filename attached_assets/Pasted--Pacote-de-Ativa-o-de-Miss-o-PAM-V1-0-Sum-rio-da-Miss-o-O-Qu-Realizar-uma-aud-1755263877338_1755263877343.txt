**### Pacote de Ativação de Missão (PAM) V1.0**

* **Sumário da Missão (O Quê):** Realizar uma auditoria forense completa da "Tela de Cobranças" existente, comparando sua implementação atual com o novo "Blueprint de Negócio" definido pelo Arquiteto.
* **Intenção Estratégica (O Porquê):** Antes de refatorar, precisamos de um mapa preciso de todas as lacunas entre o que o sistema faz hoje e o que ele *deveria* fazer. Esta auditoria nos dará o plano de ataque para a fase de implementação.
* **Histórico Relevante (Consulta Obrigatória):** A diretriz do Arquiteto é nossa nova fonte da verdade. Ignore todas as implementações de filtro anteriores. Sua missão é avaliar o código atual contra as novas regras absolutas (ex: "aparecer apenas com `BOLETOS_EMITIDOS`").
* **Modelo Mental (Como se Encaixa):** Você é um auditor de sistemas. Sua tarefa é ler, analisar e relatar as discrepâncias entre a especificação (o Blueprint) e a realidade (o código).
* **Riscos Antecipados:** **Risco:** Uma análise superficial levará a uma correção incompleta. **Contramedida:** A sua missão não é resolver, mas sim **PROVAR** onde o sistema falha, apresentando o código como evidência.

**CONSTRAINTS (Roadmap de Auditoria Forense):**

**1. Auditoria da "Regra de Entrada" (A Query Principal):**
    - **Ação:** Localize a query Drizzle no endpoint `GET /api/cobrancas`.
    - **Análise Crítica:** Compare a cláusula `WHERE` atual com a nova regra de negócio: a query está a filtrar propostas cujo `status` seja `BOLETOS_EMITIDOS` (ou posterior)? Ou ainda usa a lógica antiga de flags booleanas?
    - **Relatório:** Apresente a cláusula `WHERE` atual e declare se ela está `[CONFORME]` ou `[NÃO CONFORME]` com a nova regra.

**2. Auditoria das "Informações Críticas" (O Payload da API):**
    - **Ação:** Analise a cláusula `.select()` da mesma query.
    - **Análise Crítica:** O endpoint está a retornar **TODOS** os campos necessários para a UI, conforme definido no Blueprint (incluindo dados para `Valor da Próxima Parcela`, `Data de Vencimento`, etc.)?
    - **Relatório:** Liste os campos que a API está a retornar atualmente. Aponte quaisquer "Informações Críticas" que estejam em falta no payload.

**3. Auditoria das "Ações Primárias" (Funcionalidades):**
    - **Ação:** Inspecione os endpoints e a lógica de backend para as ações de "Aplicar Desconto" e "Prorrogar Vencimento".
    - **Análise Crítica:** Estas funcionalidades estão completamente implementadas e a interagir com a API do Banco Inter, ou são apenas placeholders?
    - **Relatório:** Descreva o estado atual (funcional, parcial ou placeholder) da lógica por trás destes botões de ação.

**Relatório Final:**
O critério de sucesso é um relatório de auditoria que nos dê um mapa claro de todas as correções necessárias para alinhar a "Tela de Cobranças" com o nosso novo Blueprint de Negócio.
