**IMPLEMENTAR:** Na "Tela de Cobranças", as ações de "Prorrogar Vencimento" e "Aplicar Desconto" para boletos individuais.

**CONTEXT:** A equipe de cobranças precisa de ferramentas para gerenciar ativamente os boletos, oferecendo flexibilidade aos clientes. Estas ações devem modificar o boleto real através de uma chamada à API do Banco Inter.

**CURRENT STATE:** A tabela de cobranças é somente leitura. Não existem ações de modificação para os boletos.

**EXPECTED (Estado Final de Sucesso):**

**1. Menu de Ações:**
    - Em cada linha da tabela de cobranças, adicione uma coluna "Ações" com um menu (dropdown ou similar).
    - Este menu deve conter duas opções: "Prorrogar Vencimento" e "Aplicar Desconto".
    - **Regra de Negócio:** Estas opções só devem estar habilitadas para boletos com status que permitam modificação (ex: `A_RECEBER`, `ATRASADO`). Devem estar desabilitadas para boletos com status `PAGO` ou `CANCELADO`.

**2. Funcionalidade "Prorrogar Vencimento":**
    - Ao clicar na opção, um modal deve ser exibido.
    - O modal deve conter um campo de seleção de data (`date picker`) para o usuário escolher a `novaDataVencimento`.
    - Ao confirmar, o sistema deve fazer uma chamada ao backend, que por sua vez utilizará a função `editarCobranca` do `interBankService` para atualizar o boleto na API do Inter com a nova data.

**3. Funcionalidade "Aplicar Desconto":**
    - Ao clicar na opção, um segundo tipo de modal deve ser exibido.
    - O modal deve ter campos para o usuário definir um desconto de **valor fixo**: um campo para o `valor` do desconto e um campo de data para a `dataLimite` do desconto.
    - Ao confirmar, o sistema deve chamar o backend, que utilizará a função `editarCobranca` do `interBankService` para aplicar o desconto na API do Inter. O payload do desconto deve usar o código `VALORFIXODATAINFORMADA`.

**INTEGRAÇÃO:**
- Crie um novo endpoint no backend, por exemplo `PATCH /api/inter/collections/:codigoSolicitacao`, para receber e processar as solicitações de modificação do frontend.
- Este endpoint deve ser seguro e chamar a função `editarCobranca` do `interBankService` com o payload apropriado.

**ROLES:** Apenas usuários com a role `ADMIN` devem ter permissão para executar estas ações de modificação. Para usuários `COBRANÇA`, as opções no menu devem estar visíveis, porém desabilitadas (read-only).

**TESTE:**
- Valide que o menu "Ações" aparece e está habilitado/desabilitado conforme o status do boleto.
- Valide que usuários `COBRANÇA` não conseguem acionar as ações.
- Teste a função de prorrogar, confirme a chamada à API e verifique se a data é atualizada na UI.
- Teste a função de aplicar desconto e valide a chamada e o retorno da API.

**BANCO:** Após cada modificação bem-sucedida confirmada pela API do Inter, atualize os dados correspondentes (como `dataVencimento`) na nossa tabela local `inter_collections` para manter a consistência dos dados.
